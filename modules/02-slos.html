Module 02

SLIs, SLOs & Error Budgets

Define measurable reliability targets and use error budgets to balance shipping features with stability.

The Reliability Stack

  • SLI (Service Level Indicator) — A quantitative measure of service behavior.
  • SLO (Service Level Objective) — A target value for an SLI over a time window.
  • SLA (Service Level Agreement) — A business contract with consequences for missing targets.
  • Error budget — The allowed amount of unreliability before you must prioritize stability.

Choosing Good SLIs

SLIs should reflect what users actually experience. Common categories:

  • Availability — Did the request succeed? (e.g., HTTP 2xx / total requests)
  • Latency — How long did it take? (e.g., p99 < 300ms)
  • Throughput — Can the system handle the load?
  • Freshness — How up-to-date is the data?
Rule of thumb If you can't explain your SLI in terms of user impact, it's probably the wrong metric.

Setting SLOs

Start with user expectations, not infrastructure limits. Example for an API service:

SLI:  Proportion of successful requests (non-5xx)
SLO:  99.9% of requests succeed over a 30-day window
      (≈ 43 minutes of downtime per month)

SLO Best Practices

  • Fewer SLOs are better — focus on 2–4 per service.
  • Involve product and engineering stakeholders when setting targets.
  • Review SLOs quarterly as user expectations evolve.
  • Don't set SLOs at 100% — it blocks all change and isn't achievable.

Error Budgets

An error budget is 1 − SLO. For a 99.9% SLO, your error budget is 0.1% of requests that can fail (or ~43 minutes/month of downtime).

  • Budget remaining → Team can ship features, take risks, run experiments.
  • Budget exhausted → Freeze launches, focus on reliability work until budget recovers.
Common mistake Setting SLOs without enforcing error budget policies. An SLO nobody acts on is just a dashboard decoration.

Example: Error Budget Policy

IF error_budget_remaining < 10%:
  - No new feature launches without SRE sign-off
  - All on-call focus shifts to reliability improvements
  - Weekly review with engineering leadership

IF error_budget_remaining > 50%:
  - Normal development velocity
  - Optional: invest surplus budget in risky but valuable experiments
