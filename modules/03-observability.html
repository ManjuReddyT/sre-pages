Module 03

Observability

Build the visibility you need to detect problems, debug incidents, and validate SLOs in production.

Monitoring vs. Observability

Monitoring tells you when something is wrong based on known failure modes. Observability lets you ask arbitrary questions about system behavior you didn't anticipate. Both are essential — monitoring catches known issues fast; observability helps you investigate the unknown.

The Three Pillars

1. Metrics

Numeric time-series data. Best for dashboards, alerting, and SLO tracking.

  • RED method (services): Rate, Errors, Duration
  • USE method (resources): Utilization, Saturation, Errors
  • Popular stacks: Prometheus + Grafana, Datadog, CloudWatch

2. Logs

Discrete events with context. Essential for debugging and audit trails.

  • Use structured logging (JSON) for machine-parseable output
  • Include correlation IDs to trace requests across services
  • Aggregate with ELK, Loki, or cloud-native log services

3. Traces

End-to-end request paths across distributed systems.

  • Show latency breakdown per service and span
  • Identify bottlenecks in microservice architectures
  • Tools: Jaeger, Zipkin, OpenTelemetry, Honeycomb
OpenTelemetry The industry standard for instrumenting apps once and exporting to any backend. Start here for new projects.

Alerting That Works

Bad alerts cause alert fatigue. Good alerts are:

  • Actionable — Someone can do something about it right now.
  • User-facing — Tied to SLOs or symptoms users would notice.
  • Symptom-based — Alert on "checkout failing" not "CPU > 80%".
  • Rare — If everything pages, nothing pages.

Example: Golden Signals Dashboard

Service: payment-api
├── Request rate (req/s)
├── Error rate (% 5xx)
├── Latency (p50, p95, p99)
├── Saturation (CPU, memory, connection pool)
└── SLO burn rate (1h / 6h windows)
Anti-pattern Alerting on low-level infrastructure metrics without connecting them to user impact. Disk 70% full is not an emergency unless users are affected.
