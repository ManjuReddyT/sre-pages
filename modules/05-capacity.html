Module 05

Capacity & Performance

Plan ahead for growth, test under load, and avoid the outage that "came out of nowhere."

Why Capacity Planning Matters

Most scalability incidents aren't sudden — they're predictable. Traffic grows, data accumulates, and one day the system hits a cliff. Capacity planning turns "surprise outage" into "scheduled scaling event."

Capacity Planning Process

  1. Measure current usage — CPU, memory, requests/sec, storage growth rate.
  2. Forecast demand — Use historical trends + business projections (launches, marketing events).
  3. Identify bottlenecks — Database connections, API rate limits, disk I/O, network bandwidth.
  4. Plan headroom — Target 30–50% spare capacity for normal ops; more before known spikes.
  5. Load test — Validate assumptions before production proves you wrong.

Load Testing

Test in staging with production-like data volumes. Key practices:

  • Start small, ramp gradually — find the breaking point methodically
  • Test realistic traffic patterns, not just peak RPS
  • Include failure scenarios (dependency timeouts, partial outages)
  • Tools: k6, Gatling, Locust, JMeter
# Example k6 smoke test
import http from 'k6/http';
import { check } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },
    { duration: '5m', target: 100 },
    { duration: '2m', target: 0 },
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],
    http_req_failed: ['rate<0.01'],
  },
};

export default function () {
  const res = http.get('https://api.example.com/health');
  check(res, { 'status is 200': (r) => r.status === 200 });
}

Performance Optimization Priorities

  1. Fix the slowest bottleneck first (Amdahl's Law)
  2. Cache aggressively at every layer (CDN, app, database)
  3. Scale horizontally before vertically
  4. Optimize the database — most apps are I/O bound
Don't premature-optimize Measure first. A profiler and a load test beat guessing every time.
