MODULE 1

Introduction to strace

What is strace?

strace is a powerful diagnostic, debugging, and instructional userspace utility for Linux. It is used to monitor and tamper with interactions between user-space processes and the Linux kernel, which are known as system calls (syscalls).

Why was strace created?

Before strace, debugging complex applications involved tedious methods like attaching GDB or relying solely on application logs. strace was designed to provide a clear, concise view of how a program interacts with the kernel without modifying code.

How ptrace works internally

strace leverages the ptrace system call. It allows one process (tracer) to observe and control another (tracee), intercept system calls, examine registers and memory.

Key Insight

When strace attaches to a process, it can stop the tracee before and after each system call, record the syscall number, arguments, and return value.

Syscall Lifecycle

  1. Initiation via syscall instruction
  2. Mode switch: user → kernel
  3. Kernel executes the service
  4. Return value placed in register
  5. Mode switch back to user space

Blocking vs Non-Blocking Syscalls

Blocking: Process pauses until operation completes (e.g. read on empty pipe).

Non-Blocking: Returns immediately with EAGAIN if data not ready. Used with epoll/select for high concurrency.

MODULE 3

strace Fundamentals

Core Commands

strace -p <PID> Attach to running process
strace <command> Start new process under trace
strace -f Follow child processes & threads
strace -ff -o prefix Separate file per thread

Timing & Output

-t — Time of day
-tt — Time with microseconds
-T — Duration of each syscall
-c — Summary counts & time
-o file — Write to file instead of stderr
MODULE 4

Understanding Common System Calls

File System

  • open / openat — ENOENT, EACCES, EISDIR
  • read / write — blocking, EAGAIN, EPIPE
  • stat / lstat / access — permission & existence checks

Network

  • socket + connect — ECONNREFUSED, ETIMEDOUT, EINPROGRESS
  • accept, send, recv, recvfrom
  • epoll_wait / poll / select — event driven I/O
SRE TIP

A process stuck in futex(..., FUTEX_WAIT, ...) without return value is almost always a deadlock or severe lock contention.

MODULE 10

Deep Dive into futex

futex is the kernel primitive behind all high-level locks in pthreads, Java, Go, etc. High volume of FUTEX_WAIT + FUTEX_WAKE on the same address = lock contention hot spot.

MODULE 11

Java Application Debugging with strace

Combine strace with jstack, JFR, and GC logs. When you see many threads in futex waits, use jstack to find which Java lock they are blocked on.

MODULE 13

strace vs Other Tools

Decision tree: Use strace when you need ground-truth syscall visibility. Use perf/eBPF for lower-overhead CPU profiling. Use jstack/JFR first for Java. Use tcpdump for packet-level network truth.

MODULE 14

Production Best Practices

  • Always filter aggressively with -e and -P
  • Redirect output with -o and use timeout
  • Prefer ephemeral containers in Kubernetes
  • Never trace sensitive processes without strict controls
  • Start with higher-level tools; reach for strace when OS boundary is suspected

strace Production Cheat Sheet

Print-friendly quick reference

Core Commands

strace -p <PID> — Attach to running process
strace -f -p <PID> — Follow forks & threads
strace -T -p <PID> — Show syscall durations
strace -c -p <PID> — Summary report
strace -e trace=network — Network only
strace -P /path/to/file — Specific path only

Common Patterns

futex(..., FUTEX_WAIT, ...) — HANG / DEADLOCK
recvfrom(...) [no return] — Network wait
stat() = -1 ENOENT — Busy loop / missing file
connect() = -1 ECONNREFUSED — Service not listening
openat() = -1 EACCES — Permission denied
bind() = -1 EADDRINUSE — Port conflict
